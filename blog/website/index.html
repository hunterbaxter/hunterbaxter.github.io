    • Motivation

    Not using mainstream social media (Facebook, Twitter, Snapchat, Instagram) has often made keeping in touch with others difficult. A personal website allows me to give updates to my life publicly in a way that social media does not (maybe I will write a post one day). In addition, it will help me professionally to have a centralized location for my public work.

    The motivation to create my site rather than use a theme was to have the specific website I had in mine without any compromises and the knowledge to fix the errors that will inevitably occur. This article broadly describes a subset of the choices and technical details that went into making my website.

    Planning

    Prior Sites, Frameworks, and Framework choice

    In the past, I have used the Jekyll theme Academic Pages and Hugo PaperMod. The reason I was not satisfied with these themes was:

    1. They were very mature, as a result, complicated. Since I am very inexperienced with front-end development, I found tweaking the sites thorny.
    2. The git repositories were outdated with the current version of {Jekyll, Hugo} when I was using it - breaking some features of the site (which I did not know how to fix)

    Studying the source code and learning the frameworks would mitigate the issue of not understanding the complexities of the themes, but instead, I chose to start from scratch (which has the benefit of learning basic HTML and CSS).

    Since I started from scratch, I had to pick a static site generator. Static site generators essentially all do the same thing, and unless you are doing something custom, choose one with a theme you like and has markdown rendering. However, I am doing something custom from scratch, so the decision would heavily impact my enjoyment of the process of building the site. I chose Zola because it is written in a language I know well, has fast live updates, and has a templating language like Jinja2.

    Wireframes and Site Design

    In one of my classes at university, I remember working on a project that had a client, and our front-end developer made wireframes before any development. The rationality was that if there is a clear idea of the site/look beforehand, progress moves in a clear direction with well-defined milestones. Since I am the client, I did not put a ton of effort into creating a polished wireframe and instead sketched my ideas onto a tablet quickly.

    • Header and Footer: For the header, it was important to me that it be responsive and collapse when viewed on a smaller screen. I will describe the responsive aspect in the technical section.
    • Home Page: I haven't implemented this yet (the rest of the site was enough work), as I don't have enough pictures I am comfortable using publicly, but the idea of having a more creative homepage would make my site a little more non-tech user appealing. Maybe one day, but not today. So, for now, the home page redirects to the about page.
    • About and Contact Pages: The about page gives a short bio about me, and maybe later, it can have some additional sections that go into more details about a particular area of my life. It might also contain instructions for getting in contact with me or collaborating.
    • Professional Pages: Once I have built up a public portfolio I am comfortable displaying publicly; I will have separate pages of publications and software with slightly different requirements.
    • Blog Pages: For more informal content, I wanted to have a simple card design and the option to filter posts by tags. The nice thing about Zola is that each tag can have a unique RSS feed.

    Aesthetics

    Although unpopular in a digital format, I enjoy greyscale color palettes (with an occasional exception for code snippets). I also prefer the minimal designs. So, I went with simple higher contrast colors (#F5F5F5), clean designs, and my personal font family of choice: IBM Plex (specifically Sans 1). I believe this decision is sound because most users can and should (when possible) use reader view on their browser, which lets them control font, page color, and text size (some even have text-to-speech). So, my site aesthetic choices are just an opinionated default.

    Technical Aspects

    I wanted my site to be responsive and mobile-first, as more and more people are viewing the web through mobile devices like tablets (again, people should use reader view, but I wanted to prepare for a less tech-savvy audience). Bootstrap is the most popular framework for responsive design, and AdiDoks, a theme I drew inspiration and code from, uses it. I was pleased with using Bootstrap, as there were countless examples online, and the documentation was manageable.

    Header

    The header of my site is a very simple bootstrap navbar. The notable features reside in the navbar-toggler class, where the navbar collapses when on a mobile device or small screen device. The collapsable header greatly enhanced the cellphone experience when testing.

    {% macros navbar() %} defines a macro (just like a shortcode), allowing me to reuse the code on multiple pages without duplication.

    {% macro navbar() %}
<nav class="navbar navbar-expand-sm navbar-light">
  <div class="container">
    <a class="navbar-brand" href="/">{{ config.title }}</a>
    <button class="navbar-toggler" type="button" data-bs-toggle="collapse" data-bs-target="#toggleMobileMenu" aria-controls="toggleMobileMenu" aria-expanded="false" aria-label="Toggle navigation">
      <span class="navbar-toggler-icon"></span>
    </button>
    <div class="collapse navbar-collapse" id="toggleMobileMenu">
      <ul class="navbar-nav ms-auto text-center">
        <li class="navbar-item"><a class="nav-link" href="/menu_section/"> about </a></li>
        <!-- insert more sections -->
      </ul>
    </div>
  </div>
</nav>
{% endmacro %}

    Responsive Text

    Although I always encourage people to use the reader view on their browser (I must have said this a couple of times by now haha), some may not know it exists. So, I need to account for this and make sure all of my text content looks good on any screen size. Bootstrap uses a Grid System, and rather than trying to explain, I encourage readers to view the link (I am not very knowledgeable on it and couldn't explain it well). The below code snippet contains much of the common code that wraps many elements on my site.

    {% block content %}
<div class="container">
    <div class="row justify-content-center">
      <div class="col-md-12 col-lg-10 col-xxl-8">
        <h1>{{ page.title }}</h1>
        <p>{{ page.content | safe }}</p>
      </div>
    </div>
</div>
{% endblock content %}

    Table of Contents

    I wouldn't say this was a complicated section to implement in HTML, as the Zola Documentation gives the exact necessary code, but I found that it didn't look good unless inserted into a Bootstrap Accordion component with some CSS to make it pleasing to the eyes. In addition, there were some annoying aspects of the accordion component when focused, so it took some digging to write the CSS to remove them. This video in particular was a good resource.

    {% macro page_toc(page) %}
{% if page.extra.toc %}
<!-- use accordion-flush to get rid of border -->
<div class="accordion" id="accordionToc">
  <div class="accordion-item">
    <h2 class="accordion-header" id="headingToc">
      <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#collapseToc" aria-expanded="false" aria-controls="collapseToc">
        Table of Contents
      </button>
    </h2>
    <div id="collapseToc" class="accordion-collapse collapse" aria-labelledby="headingToc" data-bs-parent="#accordionToc">
      <div class="accordion-body">
        {% for h1 in page.toc %}
          <li><a href="{{ h1.permalink | safe }}"> {{ h1.title }}</a></li>
    	  {% if h1.children %}
    	  <ul>
    	    {% for h2 in h1.children %}
    	    <li><a href="{{ h2.permalink | safe }}">{{ h2.title }}</a></li>
    	    {% endfor %}
    	  </ul>
    	  {% endif %}
    	  {% endfor %}
      </div>
    </div>
  </div>
</div>
{% endif %}
{% endmacro %}

    .accordion-button {
  background: $light-background-color;
  active-background: $light-background-color
}
.accordion-button:not(.collapsed) {
    color: inherit;
    background: $light-background-color
}
.accordion-button:focus {
    box-shadow: inherit;
}
// changes arrow color to black
.accordion-button:not(.collapsed)::after {
    filter: brightness(0%)
}
// actual drop down color
.accordion-item {
  background: $light-background-color;
}

    Technical Challenges

    • For some reason, markdown rendering has strange issues (tables, blockquotes) on my site, but no other example sites. Peculiar issues lead me to think I did something wrong, but it is plausible it is Zola's fault (unlikely). However, I tested markdown on barebones Zola and Hugo sites, and issues only occur on Zola. I created a bit of custom CSS to mitigate the issues with blockquotes (for now, I haven't dealt with tables).
    blockquote {
  margin-top: 10px;
  margin-bottom: 10px;
  padding-left: 15px;
  border-left: 3px solid $fu-purple; // fu is furman university
}

    For footnotes, it also does not have redirects. Maybe Zola will add or fix these features in the future (they don't bother me enough for me to implement).

    • I foresee having mild issues in the future with the depreciation of Bootstrap, and then I need to make a bunch of small annoying changes to upgrade to Bootstrap-{5+n}. Hopefully, this is in 5+ years because I don't want to touch front-end source code for a long time (HTML and CSS are mind-numbing).

    Hosting

    Currently, I am using Github pages, but I plan to get a domain name and use another hosting service in the future. However, for now, I can't justify the cost.

    Conclusion

    I am happy with how the site turned out. It took significantly longer than expected, with many frustrating moments (particularly with CSS), but I think it was worth it to have my ideal public digital presence.
